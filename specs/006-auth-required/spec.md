## User Scenarios & Testing *(mandatory)*

### User Story 1 - 비로그인 사용자 접근 차단 (Priority: P1)

비로그인 상태의 사용자가 게시글 작성, 별주기, 댓글 작성 등 모든 서비스 기능을 이용하려 할 때 로그인 페이지로 안내된다. 로그인하지 않으면 어떤 기능도 사용할 수 없다.

**Why this priority**: 회원 전용 전환의 핵심이다. 비회원 접근 차단 없이는 나머지 변경이 의미가 없다.

**Independent Test**: 로그아웃 상태에서 게시글 작성, 별주기, 댓글 작성을 시도했을 때 모두 로그인 페이지로 리다이렉트되는 것을 확인한다.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 게시글 작성을 시도할 때, **When** 작성 버튼을 클릭하면, **Then** 로그인 페이지로 리다이렉트된다
2. **Given** 비로그인 사용자가 별주기를 시도할 때, **When** 별주기 버튼을 클릭하면, **Then** 로그인 페이지로 리다이렉트된다
3. **Given** 비로그인 사용자가 직접 API를 호출할 때, **When** 인증 토큰 없이 요청하면, **Then** 인증 실패 응답이 반환된다

---

### User Story 2 - 닉네임 입력 제거 및 사용자 정보 자동 연결 (Priority: P1)

기존에 비회원용으로 제공되던 닉네임 입력 필드가 제거되고, 로그인한 사용자의 이름이 자동으로 게시글/별/댓글에 연결된다. 사용자는 별도 닉네임 입력 없이 바로 기능을 이용할 수 있다.

**Why this priority**: 비회원 접근 차단과 함께 기존 닉네임 기반 로직을 회원 기반으로 전환해야 일관된 시스템이 된다.

**Independent Test**: 로그인 사용자가 게시글 작성, 별주기를 할 때 닉네임 입력 없이 자동으로 사용자 이름이 표시되는 것을 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인 사용자가 게시글을 작성할 때, **When** 제목과 내용만 입력하고 제출하면, **Then** 게시글에 사용자 이름이 자동으로 표시된다
2. **Given** 로그인 사용자가 별을 줄 때, **When** 별주기 버튼만 클릭하면, **Then** 닉네임 입력 없이 즉시 별이 추가되고 사용자 이름이 연결된다
3. **Given** 기존에 닉네임으로 작성된 게시글/별이 있을 때, **When** 해당 콘텐츠를 조회하면, **Then** 기존 닉네임이 그대로 표시된다 (하위 호환)

---

### User Story 3 - 비로그인 사용자 열람 허용 범위 결정 (Priority: P2)

비로그인 사용자도 3D 공간(은하, 행성)을 탐색하고 게시글 내용을 읽는 것은 가능하다. 다만, 어떤 쓰기 작업(게시글 작성, 별주기, 댓글)도 할 수 없으며, 쓰기 시도 시 로그인 유도 메시지가 표시된다.

**Why this priority**: 읽기 접근까지 차단하면 신규 사용자 유입이 어려워진다. 열람은 허용하되 쓰기만 차단하는 것이 커뮤니티 성장에 유리하다.

**Independent Test**: 비로그인 사용자가 은하/행성을 탐색하고 게시글을 읽을 수 있지만, 작성 관련 UI가 비활성화되어 있는 것을 확인한다.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 사이트에 접속할 때, **When** 3D 공간을 탐색하면, **Then** 은하와 행성을 자유롭게 둘러볼 수 있다
2. **Given** 비로그인 사용자가 게시글 상세를 열 때, **When** 행성을 클릭하면, **Then** 게시글 내용을 읽을 수 있지만 작성/별주기/댓글 UI는 비활성화되고 "로그인이 필요합니다" 안내가 표시된다

---

### Edge Cases

- 기존 닉네임 기반 게시글/별 데이터는 어떻게 되는가? → 기존 데이터의 닉네임 필드는 그대로 유지하여 하위 호환 보장
- 로그인 세션이 만료된 상태에서 작성 중이던 내용은? → 제출 시점에 인증 확인 후 로그인 페이지로 안내, 작성 내용은 유실될 수 있음
- 기존 optional auth 가드가 적용된 엔드포인트는? → 모든 쓰기 엔드포인트의 가드를 필수 인증으로 전환
- 닉네임 필드가 제거된 후 기존 데이터에 authorId가 null인 게시글은? → null authorId는 "탈퇴한 사용자" 또는 "알 수 없음"으로 표시

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 모든 쓰기 작업(게시글 작성, 별주기, 댓글 작성)은 로그인한 회원만 수행할 수 있어야 한다
- **FR-002**: 비로그인 사용자가 쓰기 작업을 시도하면 로그인 페이지로 안내되어야 한다
- **FR-003**: 기존 닉네임 입력 필드를 제거하고, 로그인 사용자의 이름이 자동으로 연결되어야 한다
- **FR-004**: 비로그인 사용자도 3D 공간 탐색 및 게시글 열람(읽기)은 가능해야 한다
- **FR-005**: 비로그인 사용자에게 쓰기 관련 UI는 비활성화 상태로 표시되고 로그인 유도 메시지가 노출되어야 한다
- **FR-006**: 기존에 닉네임으로 작성된 게시글, 별 데이터는 하위 호환되어 기존 닉네임이 그대로 표시되어야 한다
- **FR-007**: 모든 쓰기 API 엔드포인트는 인증 토큰이 없는 요청을 거부해야 한다
- **FR-008**: 별주기 시 닉네임 입력 단계가 제거되고, 버튼 클릭만으로 즉시 별이 추가되어야 한다
- **FR-009**: 한 사용자는 동일 행성에 별을 1개만 줄 수 있다 (중복 별주기 불가). 이미 별을 준 행성에서는 별주기 버튼이 비활성화되고 "이미 별을 주었습니다" 안내가 표시되어야 한다

### Key Entities

- **User (사용자)**: 기존 User 엔티티. 모든 쓰기 작업의 주체로서, 게시글/별/댓글과 필수 관계를 맺음 (기존 optional → required로 전환)
- **Planet (게시글)**: 기존 authorNickname 필드는 유지하되, 새 게시글은 반드시 author(User)와 연결
- **Star (별)**: 기존 giverNickname 필드는 유지하되, 새 별은 반드시 giver(User)와 연결

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 비로그인 사용자의 쓰기 작업 성공률이 0%이다 (모든 쓰기 시도가 차단됨)
- **SC-002**: 로그인 사용자가 닉네임 입력 없이 별주기를 1회 클릭으로 완료할 수 있다
- **SC-003**: 비로그인 사용자가 3D 공간 탐색 및 게시글 열람을 지장 없이 이용할 수 있다
- **SC-004**: 기존 닉네임 기반 게시글/별 데이터가 100% 정상 표시된다 (하위 호환)
- **SC-005**: 비로그인 사용자에게 쓰기 UI 접근 시 로그인 안내가 100% 노출된다

## Assumptions

- 기존 Clerk 인증 시스템을 그대로 활용하며, 인증 방식 자체는 변경하지 않음
- 읽기(GET) 엔드포인트는 인증 없이 접근 가능하게 유지
- 기존 닉네임 필드(authorNickname, giverNickname)는 DB에서 삭제하지 않고, 새 데이터에서는 사용하지 않음
- 005-satellite-comments(댓글 기능)에서도 동일한 회원 전용 정책이 적용됨
- OptionalClerkAuthGuard를 사용하는 모든 쓰기 엔드포인트를 ClerkAuthGuard(필수)로 교체
