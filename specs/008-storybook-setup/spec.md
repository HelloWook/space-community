## User Scenarios & Testing _(mandatory)_

### User Story 1 - 컴포넌트 독립 렌더링 및 상태 확인 (Priority: P1)

개발자가 스토리북 환경에서 프로젝트의 UI 컴포넌트(버튼, 입력, 카드, 오버레이 등)를 앱 실행 없이 독립적으로 렌더링하고, 각 컴포넌트의 다양한 상태(기본, 호버, 비활성, 에러, 로딩 등)를 시각적으로 확인할 수 있다.

**Why this priority**: 리디자인 작업 중 컴포넌트 변경의 시각적 결과를 빠르게 검증하는 것이 생산성의 핵심이다. 앱 전체를 실행하고 특정 화면까지 탐색하는 비용을 제거한다.

**Independent Test**: 스토리북 서버를 실행하고 특정 컴포넌트 스토리를 열어 정상 렌더링되는지, 상태 전환이 동작하는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 개발자가 스토리북 서버를 시작하면, **When** 브라우저에서 스토리북에 접속하면, **Then** 등록된 모든 컴포넌트 스토리가 좌측 사이드바에 카테고리별로 정리되어 표시된다
2. **Given** 특정 컴포넌트 스토리(예: 버튼)를 선택하면, **When** 스토리가 로드되면, **Then** 해당 컴포넌트가 격리된 캔버스에 정상 렌더링되고 인터랙션(클릭, 호버)이 동작한다
3. **Given** 컴포넌트에 여러 상태 스토리(기본, 비활성, 에러 등)가 등록되어 있을 때, **When** 각 상태 스토리를 전환하면, **Then** 해당 상태에 맞는 스타일과 동작이 정확히 표현된다
4. **Given** 개발자가 컴포넌트 소스 코드를 수정하면, **When** 파일을 저장하면, **Then** 스토리북이 핫 리로드되어 변경사항이 즉시 반영된다

### User Story 2 - 디자인 토큰 정의 및 시각적 문서화 (Priority: P1)

개발자/디자이너가 프로젝트의 디자인 토큰(색상 팔레트, 타이포그래피 스케일, 간격, 그림자, 테두리 반경 등)을 체계적으로 정의하고, 스토리북 내 전용 페이지에서 한눈에 확인할 수 있다. 우주 테마의 디자인 시스템을 시각적으로 문서화하여 팀 내 일관된 디자인 의사결정을 지원한다.

**Why this priority**: 디자인 토큰이 정의되지 않으면 색상, 간격, 폰트 크기 등이 컴포넌트마다 제각각이 되어 시각적 일관성이 깨진다. 모든 UI 작업의 기초이므로 최우선으로 정의해야 한다.

**Independent Test**: 스토리북의 디자인 토큰 페이지에 접속하여 색상, 타이포그래피, 간격 값이 시각적으로 표시되고, 실제 앱에 적용된 값과 일치하는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 스토리북이 실행 중일 때, **When** 사이드바에서 "디자인 토큰" 카테고리를 열면, **Then** 색상 팔레트, 타이포그래피 스케일, 간격, 테두리 반경 등의 토큰 문서 페이지가 존재한다
2. **Given** 색상 팔레트 페이지를 열면, **When** 페이지가 로드되면, **Then** 우주 테마 색상(배경 검정, 악센트 연보라, 오버레이 배경, 글로우 보라, 텍스트 색상, 경고/성공/위험 색상)이 색상 견본(swatch)과 함께 CSS 변수 이름/HSL 값이 나열된다
3. **Given** 타이포그래피 페이지를 열면, **When** 페이지가 로드되면, **Then** 폰트 패밀리, 크기 단계(xs~4xl), 행간, 굵기가 실제 렌더링 예시와 함께 표시되고, WCAG 최소 기준(본문 16px 이상)을 만족한다
4. **Given** 간격(spacing) 페이지를 열면, **When** 페이지가 로드되면, **Then** 4px 단위 기반 간격 스케일(4, 8, 12, 16, 20, 24, 32, 48, 64)이 시각적 블록으로 표시된다

### User Story 3 - 타이포그래피 및 폰트 설정 (Priority: P1)

프로젝트 전체에 일관된 폰트 패밀리와 타이포그래피 스케일이 적용된다. 본문 텍스트는 16px 이상으로 설정되어 가독성과 웹 접근성을 보장한다. 모든 텍스트 크기가 체계적인 스케일을 따른다.

**Why this priority**: 현재 폰트가 너무 작아 가독성이 떨어지는 문제가 있다. 접근성 기준(WCAG 2.1)을 충족하려면 기본 폰트 크기를 16px 이상으로 설정하고 명확한 타이포그래피 계층을 수립해야 한다.

**Independent Test**: 앱의 모든 페이지에서 본문 텍스트가 16px 이상이고, 제목/부제목/본문/캡션이 명확한 시각적 계층을 가지는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 사용자가 어떤 페이지를 방문하든, **When** 본문 텍스트를 확인하면, **Then** 최소 16px(1rem) 이상의 크기로 렌더링된다
2. **Given** 프로젝트의 타이포그래피 스케일이 정의되었을 때, **When** 제목(h1~h4), 본문, 캡션을 비교하면, **Then** 각각 명확한 크기 차이가 있어 시각적 계층 구조가 드러난다
3. **Given** 우주 테마에 적합한 폰트가 설정되었을 때, **When** 앱 전체를 확인하면, **Then** 모든 텍스트가 통일된 폰트 패밀리(기본 + 코드용 모노스페이스)로 렌더링된다
4. **Given** 오버레이, 댓글, 게시글 등 기존 UI에서, **When** 텍스트 크기를 확인하면, **Then** 기존의 11px~13px 텍스트가 최소 14px(캡션) 또는 16px(본문) 이상으로 조정되어 있다

### User Story 4 - WASD 키보드로 우주 탐험 (Priority: P1)

사용자가 은하 보드에서 WASD 키를 사용하여 우주 공간을 자유롭게 탐험할 수 있다. 화면 하단에 조작 안내(WASD: 이동, 마우스: 회전, 스크롤: 줌)가 표시되어 처음 방문한 사용자도 직관적으로 탐험을 시작할 수 있다.

**Why this priority**: "우주를 항해하는 느낌"의 핵심은 키보드를 통한 자유 이동이다. 마우스만으로는 회전/줌만 가능하지만, WASD를 추가하면 FPS 게임처럼 공간을 관통하며 탐험하는 몰입감을 제공한다.

**Independent Test**: 은하 보드에서 W/A/S/D 키를 눌러 카메라가 전진/좌/후진/우로 이동하고, 하단에 조작 안내가 표시되는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 사용자가 은하 보드에 접속하면, **When** W 키를 누르면, **Then** 카메라가 현재 바라보는 방향으로 전진한다
2. **Given** 카메라가 이동 가능한 상태일 때, **When** A/S/D 키를 누르면, **Then** 카메라가 각각 좌/후/우로 이동한다
3. **Given** WASD 이동과 마우스 회전을 동시에 사용하면, **When** W를 누르면서 마우스를 돌리면, **Then** 카메라 방향이 바뀌면서 동시에 전진하여 자연스러운 FPS 스타일 탐험이 가능하다
4. **Given** 사용자가 은하 보드에 처음 접속하면, **When** 화면을 보면, **Then** 하단에 "WASD: 이동 | 마우스 드래그: 회전 | 스크롤: 줌 | 클릭: 선택" 형태의 조작 안내가 반투명하게 표시된다
5. **Given** 조작 안내가 표시된 상태에서, **When** 사용자가 어떤 키를 누르거나 마우스를 조작하면, **Then** 조작 안내가 서서히 사라진다 (첫 인터랙션 후 페이드아웃)
6. **Given** 오버레이가 열린 상태에서, **When** WASD 키를 누르면, **Then** 카메라가 이동하지 않는다 (오버레이 입력 우선)
7. **Given** 입력 필드(input, textarea)에 포커스가 있을 때, **When** WASD 키를 누르면, **Then** 텍스트 입력이 되고 카메라는 이동하지 않는다

### User Story 5 - 웹 접근성 준수 (Priority: P1)

프로젝트 전체가 WCAG 2.1 AA 기준의 웹 접근성을 충족한다. 텍스트 대비율, 키보드 탐색, 포커스 표시, 스크린 리더 지원 등이 보장된다.

**Why this priority**: 접근성은 사용자 범위를 넓히고 법적 요구사항을 충족하는 기본 품질이다. 폰트 크기 개선과 함께 전체적인 접근성 수준을 높여야 한다.

**Independent Test**: 주요 페이지에서 브라우저 접근성 감사 도구(Lighthouse, axe)를 실행하여 WCAG 2.1 AA 위반 0건을 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 앱의 모든 텍스트를 확인하면, **When** 텍스트 색상과 배경색의 대비율을 측정하면, **Then** WCAG 2.1 AA 기준(일반 텍스트 4.5:1, 큰 텍스트 3:1) 이상이다
2. **Given** 키보드만 사용하여 앱을 탐색할 때, **When** Tab 키로 UI 요소를 순회하면, **Then** 모든 인터랙티브 요소(버튼, 입력, 링크)에 포커스가 도달하고, 포커스 링이 명확히 보인다
3. **Given** 스크린 리더가 활성화된 상태에서, **When** UI 요소를 탐색하면, **Then** 모든 버튼과 입력 필드에 적절한 레이블(aria-label 또는 visible label)이 제공된다
4. **Given** 모든 인터랙티브 요소(버튼, 링크)를 확인하면, **When** 터치 대상 크기를 측정하면, **Then** 최소 44x44px 이상의 터치/클릭 영역을 가진다

### User Story 6 - 3D 컴포넌트 스토리북 미리보기 (Priority: P2)

개발자가 3D 씬 관련 컴포넌트(행성, 인공위성, 장식 요소 등)도 스토리북에서 독립적으로 미리볼 수 있다.

**Why this priority**: 2D UI 컴포넌트보다 복잡도가 높고, 3D 렌더링 컨텍스트 설정이 필요하므로 P1 인프라가 갖춰진 후 추가한다.

**Independent Test**: 스토리북에서 3D 컴포넌트 스토리를 열어 Canvas 내에 3D 오브젝트가 렌더링되고 회전/줌이 가능한지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 스토리북의 3D 컴포넌트 카테고리를 열면, **When** 행성 컴포넌트 스토리를 선택하면, **Then** 3D Canvas 내에 행성이 렌더링되고 마우스로 회전/줌이 가능하다
2. **Given** 3D 컴포넌트 스토리가 로드된 상태에서, **When** 컴포넌트의 속성(색상, 크기 등)을 변경하면, **Then** 3D 오브젝트에 변경사항이 실시간으로 반영된다
3. **Given** 장식 요소(유성, 운석 등) 스토리를 선택하면, **When** 스토리가 로드되면, **Then** 개별 장식 요소가 애니메이션과 함께 독립적으로 렌더링된다

---

### Edge Cases

- 3D 컴포넌트 스토리에서 WebGL Canvas 컨텍스트가 올바르게 초기화/정리되지 않으면 메모리 누수가 발생할 수 있다
- 스토리북 빌드와 앱 빌드의 설정이 충돌하여 한쪽에서 동작하는 컴포넌트가 다른 쪽에서 깨질 수 있다
- 디자인 토큰 값이 변경되었는데 스토리북 문서 페이지에 반영되지 않으면 오래된 정보를 참조하게 된다 — 토큰은 단일 소스에서 자동으로 읽어야 한다
- WASD 이동 중 카메라가 씬 경계 밖으로 나가면 빈 공간만 보일 수 있다 — 이동 범위 제한 필요
- WASD 이동과 OrbitControls 회전이 동시에 동작할 때 카메라 상태가 충돌할 수 있다
- 다국어 키보드 레이아웃(AZERTY, QWERTZ 등)에서 WASD 위치가 달라질 수 있다 — 물리적 키 위치 기준으로 동작해야 한다
- 텍스트 대비율이 우주 테마(어두운 배경)에서 충분하지 않을 수 있다 — 연한 회색(#999 이하) 텍스트는 대비율 확인 필요
- 폰트 크기 증가로 기존 레이아웃(400px 오버레이 패널 등)이 깨질 수 있다

## Requirements _(mandatory)_

### Functional Requirements

**스토리북 환경**
- **FR-001**: 시스템은 독립적인 컴포넌트 개발 환경(스토리북)을 제공하여 앱 실행 없이 UI 컴포넌트를 렌더링하고 확인할 수 있어야 한다
- **FR-002**: 스토리북은 개발 의존성으로만 설치되어 프로덕션 빌드에 포함되지 않아야 한다
- **FR-003**: 스토리북은 프로젝트의 기존 스타일링 체계(Tailwind CSS, shadcn/ui 테마)를 동일하게 적용하여 앱과 일관된 렌더링 결과를 보여야 한다
- **FR-004**: 컴포넌트 소스 코드 변경 시 스토리북이 자동으로 핫 리로드되어야 한다

**컴포넌트 스토리**
- **FR-005**: 각 UI 컴포넌트(버튼, 입력, 카드, 오버레이 등)는 최소 기본 상태, 비활성 상태, 에러 상태의 스토리를 가져야 한다
- **FR-006**: 스토리는 카테고리(일반 UI, 폼, 오버레이, 3D 등)별로 사이드바에 정리되어야 한다
- **FR-007**: 각 스토리에서 컴포넌트의 속성(props)을 동적으로 변경하여 결과를 확인할 수 있어야 한다

**디자인 토큰 정의**
- **FR-008**: 시스템은 색상 토큰을 정의해야 한다 — 배경(background), 전경(foreground), 주색(primary), 악센트(accent/연보라), 오버레이 배경(overlay-bg), 글로우(glow-purple), 뮤트(muted), 파괴(destructive), 테두리(border), 입력(input), 카드(card) 등
- **FR-009**: 시스템은 타이포그래피 토큰을 정의해야 한다 — 폰트 패밀리(기본 sans-serif + 코드용 monospace), 크기 스케일(xs: 12px, sm: 14px, base: 16px, lg: 18px, xl: 20px, 2xl: 24px, 3xl: 30px, 4xl: 36px), 행간(tight: 1.25, normal: 1.5, relaxed: 1.75), 굵기(normal: 400, medium: 500, semibold: 600, bold: 700)
- **FR-010**: 시스템은 간격 토큰을 정의해야 한다 — 4px 단위 기반 스케일(1: 4px, 2: 8px, 3: 12px, 4: 16px, 5: 20px, 6: 24px, 8: 32px, 12: 48px, 16: 64px)
- **FR-011**: 시스템은 테두리 반경 토큰을 정의해야 한다 — sm: 4px, md: 8px, lg: 12px, xl: 16px, full: 9999px
- **FR-012**: 모든 디자인 토큰은 CSS 변수로 정의되어 단일 소스(globals.css)에서 관리되어야 한다

**디자인 토큰 문서**
- **FR-013**: 스토리북 내에 색상 팔레트, 타이포그래피, 간격 등 디자인 토큰을 시각적으로 보여주는 전용 문서 페이지가 존재해야 한다
- **FR-014**: 디자인 토큰 문서는 프로젝트의 실제 토큰 정의(CSS 변수)에서 자동으로 값을 읽어와야 한다 (수동 복사 금지)
- **FR-015**: 디자인 토큰 변경 시 스토리북 문서 페이지에 즉시 반영되어야 한다

**타이포그래피 및 폰트**
- **FR-016**: 본문 텍스트의 기본 크기는 16px(1rem) 이상이어야 한다
- **FR-017**: 캡션/보조 텍스트의 최소 크기는 14px 이상이어야 한다 (기존 11~13px 텍스트 모두 상향)
- **FR-018**: 프로젝트 전체에 통일된 폰트 패밀리가 적용되어야 한다 (기본 + 코드용 모노스페이스)
- **FR-019**: 제목(h1~h4)과 본문 간에 명확한 시각적 계층이 존재해야 한다

**WASD 우주 탐험**
- **FR-020**: 은하 보드에서 W/A/S/D 키를 눌러 카메라를 전진/좌/후진/우로 이동할 수 있어야 한다
- **FR-021**: WASD 이동은 카메라가 현재 바라보는 방향을 기준으로 동작해야 한다 (회전 후에도 W는 전진)
- **FR-022**: WASD 이동과 마우스 드래그 회전/스크롤 줌이 동시에 동작해야 한다
- **FR-023**: 화면 하단에 조작 안내 UI가 표시되어야 한다 — "WASD: 이동 | 마우스 드래그: 회전 | 스크롤: 줌 | 클릭: 선택"
- **FR-024**: 조작 안내는 사용자의 첫 인터랙션(키 또는 마우스) 후 페이드아웃되어야 한다
- **FR-025**: 오버레이가 열린 상태이거나 입력 필드에 포커스가 있을 때는 WASD 카메라 이동이 비활성화되어야 한다
- **FR-026**: 카메라 이동 범위에 제한이 있어 사용자가 씬 밖으로 벗어나지 않아야 한다

**웹 접근성**
- **FR-027**: 모든 텍스트는 배경과의 대비율이 WCAG 2.1 AA 기준(일반 텍스트 4.5:1, 큰 텍스트 3:1)을 충족해야 한다
- **FR-028**: 모든 인터랙티브 요소는 키보드(Tab)로 접근 가능하고 포커스 링이 명확히 보여야 한다
- **FR-029**: 모든 버튼과 입력 필드에 접근성 레이블(aria-label 또는 visible label)이 제공되어야 한다
- **FR-030**: 인터랙티브 요소의 최소 터치/클릭 영역은 44x44px 이상이어야 한다

**3D 컴포넌트 지원**
- **FR-031**: 3D 렌더링이 필요한 컴포넌트(행성, 인공위성, 장식 요소)도 스토리북에서 독립적으로 렌더링할 수 있어야 한다
- **FR-032**: 3D 컴포넌트 스토리는 자체 3D Canvas 컨텍스트를 생성하고, 스토리 전환 시 리소스를 정리해야 한다

### Key Entities

- **디자인 토큰(Design Token)**: 프로젝트 전체에서 사용되는 시각적 상수. 색상, 타이포그래피, 간격, 테두리 반경 등을 포함. CSS 변수로 정의되며 단일 소스에서 관리됨
- **타이포그래피 스케일(Typography Scale)**: 텍스트 크기, 행간, 굵기의 체계적 단계 정의. xs(12px)~4xl(36px) 범위
- **조작 안내(Controls HUD)**: 화면 하단에 표시되는 반투명 안내 UI. 첫 인터랙션 후 페이드아웃

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 스토리북 서버 실행 후 10초 이내에 브라우저에서 컴포넌트 목록이 로드된다
- **SC-002**: 등록된 모든 컴포넌트 스토리(최소 10개 이상)가 에러 없이 렌더링된다
- **SC-003**: 디자인 토큰 문서 페이지에서 색상, 타이포그래피, 간격 토큰을 한눈에 확인할 수 있고, 각 토큰의 CSS 변수명과 값이 표시된다
- **SC-004**: 프로젝트 전체에서 11~13px 크기의 텍스트가 0건이다 (최소 14px 이상)
- **SC-005**: WCAG 2.1 AA 대비율 위반이 주요 페이지에서 0건이다
- **SC-006**: 은하 보드에서 WASD 키로 카메라가 4방향으로 부드럽게 이동하고, 마우스 회전과 동시 동작한다
- **SC-007**: 은하 보드 첫 접속 시 화면 하단에 조작 안내가 표시되고, 첫 인터랙션 후 3초 이내에 사라진다
- **SC-008**: 3D 컴포넌트(행성, 장식 요소) 스토리가 스토리북 Canvas 내에서 정상 렌더링되고 인터랙션(회전/줌)이 동작한다
- **SC-009**: 스토리북 빌드가 프로덕션 앱 번들 크기에 영향을 주지 않는다

## Assumptions

- 007-frontend-redesign에서 shadcn/ui 및 Tailwind CSS v4 기반 컴포넌트 전환이 완료되어 스토리에 등록할 컴포넌트가 존재한다
- 스토리북은 프로젝트의 프론트엔드(Next.js) 소스 디렉토리 내 컴포넌트를 대상으로 한다
- 디자인 토큰은 CSS 변수(globals.css)로 정의되며 Tailwind CSS v4 설정과 연동된다
- 3D 컴포넌트 스토리는 WebGL을 지원하는 브라우저에서만 정상 동작한다
- WASD 이동은 기존 OrbitControls와 공존하며, OrbitControls의 회전/줌 기능은 유지된다
- 폰트 크기 상향으로 인한 레이아웃 조정(오버레이 패널 너비 등)은 이 피처 범위에 포함된다
- 웹 접근성 대상은 2D UI 요소에 한정하며, 3D Canvas 내부의 접근성은 별도 피처로 분리한다
- 기본 폰트로 시스템 폰트(system-ui)를 유지하되, 우주 테마에 적합한 대안 폰트 적용은 선택 사항으로 한다
