# Feature Specification: 프론트엔드 전체 리디자인

**Feature Branch**: `007-frontend-redesign`
**Created**: 2026-03-26
**Status**: Draft
**Input**: User description: "프론트엔드 전체 리디자인 — 우주 장식 요소 추가, shadcn/ui 전환, 인라인 스타일→Tailwind 마이그레이션, 오버레이 로그인 폼, 세밀한 태스크 분할"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 풍부한 우주 환경 체험 (Priority: P1)

사용자가 은하 보드에 접속하면, 기존의 공허한 검은 배경 대신 유성(meteor), 운석(asteroid), 블랙홀, 태양 등 다양한 우주 장식 요소가 배치된 생동감 있는 우주 공간을 경험한다. 이 장식 요소들은 은하와 행성 사이의 빈 공간을 채워 시각적 몰입감을 높인다.

**Why this priority**: 사용자가 가장 직접적으로 느끼는 "우주가 너무 공허하다"는 핵심 문제를 해결한다. 시각적 풍부함은 체류 시간과 재방문율에 가장 큰 영향을 미친다.

**Independent Test**: 은하 보드 진입 시 배경에 유성, 운석, 블랙홀, 태양 등 최소 4종의 장식 요소가 렌더링되고 자연스럽게 움직이는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 사용자가 은하 보드(우주 뷰)에 접속하면, **When** 씬이 로드 완료되면, **Then** 배경에 별 파티클 필드(starfield)가 표시되어 깊이감을 준다
2. **Given** 우주 뷰가 로드된 상태에서, **When** 사용자가 화면을 관찰하면, **Then** 유성이 불규칙한 간격으로 화면을 가로지르며 꼬리(trail) 효과와 함께 이동한다
3. **Given** 은하 뷰(특정 은하 내부)에 진입하면, **When** 행성들 사이 공간을 보면, **Then** 크기가 다양한 운석(asteroid)들이 느린 속도로 궤도를 따라 떠다닌다
4. **Given** 우주 뷰에서, **When** 특정 영역을 보면, **Then** 블랙홀 오브젝트가 주변 빛을 왜곡하는 시각 효과와 함께 존재한다
5. **Given** 은하 뷰 또는 우주 뷰에서, **When** 씬이 렌더링되면, **Then** 태양(발광체) 오브젝트가 주변을 밝히는 조명 효과와 함께 배치되어 있다
6. **Given** 장식 요소들이 모두 렌더링된 상태에서, **When** 사용자가 카메라를 회전/줌하면, **Then** 장식 요소들이 3D 공간에서 자연스러운 시차(parallax) 효과를 보여준다

### User Story 2 - shadcn/ui 기반 일관된 UI 컴포넌트 (Priority: P1)

모든 UI 컴포넌트(버튼, 입력 필드, 카드, 다이얼로그 등)가 shadcn/ui 컴포넌트로 교체되어 일관된 디자인 시스템을 갖춘다. 사용자는 모든 인터랙션에서 통일된 시각 언어를 경험한다.

**Why this priority**: 디자인 시스템 기반이 먼저 갖춰져야 이후 모든 UI 작업(로그인 폼, 오버레이 등)이 일관된 스타일로 구축된다. 다른 스토리의 전제 조건이다.

**Independent Test**: 기존 인라인 스타일 버튼/입력/카드 컴포넌트를 shadcn/ui 컴포넌트로 교체한 뒤, 기존과 동일한 기능이 동작하는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 게시글 작성 폼(CreatePostForm)을 열면, **When** 입력 필드와 버튼을 보면, **Then** shadcn/ui의 Input, Button, Textarea 컴포넌트로 렌더링된다
2. **Given** 댓글 작성/목록 UI를 보면, **When** 카드와 버튼을 확인하면, **Then** shadcn/ui의 Card, Button 컴포넌트로 통일되어 있다
3. **Given** 행성 커스터마이즈 패널을 열면, **When** 옵션 선택 UI를 보면, **Then** shadcn/ui의 Select, RadioGroup, Slider 등 적절한 컴포넌트로 구성되어 있다
4. **Given** 어떤 UI 컴포넌트를 사용하든, **When** 호버/포커스/액티브 상태를 테스트하면, **Then** shadcn/ui의 일관된 인터랙션 피드백(색상, 트랜지션)이 적용된다

---

### User Story 3 - 인라인 스타일 제거 및 Tailwind CSS 전환 (Priority: P1)

기존 컴포넌트에 산재한 인라인 스타일(`style={{...}}`)이 모두 Tailwind CSS 유틸리티 클래스로 전환된다. 코드 유지보수성과 스타일 일관성이 향상된다.

**Why this priority**: shadcn/ui 전환과 동일 우선순위. 인라인 스타일이 남아 있으면 shadcn/ui 테마와 충돌하고 유지보수가 어려워진다. 병행 진행이 필요하다.

**Independent Test**: 각 컴포넌트에서 `style=` 속성을 검색하여 0건인지 확인하고, 기존 UI와 시각적으로 동일한 결과가 렌더링되는지 비교하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** GalaxyScene 컴포넌트 코드를 보면, **When** 인라인 스타일을 검색하면, **Then** 3D Canvas 자체 설정을 제외한 모든 UI 요소에 Tailwind 클래스만 사용되어 있다
2. **Given** PostOverlay 컴포넌트를 보면, **When** 패널/버튼/텍스트 스타일을 확인하면, **Then** 모든 레이아웃과 색상이 Tailwind 유틸리티 클래스로 표현되어 있다
3. **Given** CommentList, WriteCommentForm 등 댓글 관련 컴포넌트를 보면, **When** 스타일링 방식을 확인하면, **Then** 카드 레이아웃, 인덴트, 보더 등이 Tailwind 클래스로 구현되어 있다
4. **Given** CustomizePanel을 보면, **When** 컨트롤 버튼과 옵션 스타일을 확인하면, **Then** Tailwind 클래스가 적용되어 있다

---

### User Story 4 - 우주 테마 오버레이 로그인 폼 (Priority: P2)

사용자가 로그인이 필요할 때, 별도 페이지로 이동하지 않고 현재 우주 화면 위에 연보라(soft purple) 계열의 반투명 오버레이 폼이 표시된다. 로그인/회원가입이 한 화면 내에서 완료되어 몰입감이 유지된다.

**Why this priority**: P1 인프라(shadcn/ui, Tailwind)가 갖춰진 후에 구현해야 일관된 디자인이 가능하다. 사용자 경험에 큰 영향을 미치지만 기반 작업에 의존한다.

**Independent Test**: 비로그인 상태에서 로그인 트리거(버튼 클릭 등) 시 오버레이가 우주 배경 위에 표시되고, 로그인 완료 후 오버레이가 닫히는지 확인하여 테스트 가능

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 은하 보드를 보고 있을 때, **When** 로그인 버튼을 클릭하면, **Then** 현재 우주 화면이 유지된 채 연보라 계열 반투명 오버레이가 부드럽게 나타난다
2. **Given** 로그인 오버레이가 표시된 상태에서, **When** 소셜 로그인(Google/GitHub) 버튼을 클릭하면, **Then** OAuth 플로우가 진행되고 완료 시 오버레이가 자동으로 닫힌다
3. **Given** 오버레이가 표시된 상태에서, **When** 오버레이 바깥 영역(배경)을 클릭하거나 ESC를 누르면, **Then** 오버레이가 닫히고 원래 우주 화면으로 돌아간다
4. **Given** 모바일 화면에서, **When** 로그인 오버레이가 표시되면, **Then** 화면 크기에 맞게 반응형으로 조정되어 표시된다

---

### User Story 5 - 공통 오버레이 디자인 시스템 (Priority: P2)

로그인 폼뿐 아니라, 게시글 작성, 은하 생성 등 모든 오버레이/모달이 동일한 연보라 테마의 공통 오버레이 컴포넌트를 사용한다. 디자인 일관성과 코드 재사용성이 확보된다.

**Why this priority**: 로그인 오버레이와 병행하여 공통화하면 이후 모든 모달/오버레이에 즉시 적용 가능하다. 장기적 유지보수 효율을 높인다.

**Independent Test**: 게시글 작성 오버레이와 로그인 오버레이가 동일한 공통 오버레이 컴포넌트를 사용하는지 코드 검증 및 시각적 일관성 비교로 테스트 가능

**Acceptance Scenarios**:

1. **Given** 어떤 오버레이(로그인, 게시글 작성, 은하 생성)를 열든, **When** 오버레이 배경과 컨테이너를 보면, **Then** 동일한 연보라 반투명 배경과 글래스모피즘 스타일 컨테이너가 사용된다
2. **Given** 공통 오버레이 컴포넌트가 있을 때, **When** 새로운 오버레이가 필요하면, **Then** 공통 컴포넌트를 import하여 내부 콘텐츠만 교체하면 동일한 디자인이 적용된다
3. **Given** 오버레이가 열린 상태에서, **When** 열기/닫기 애니메이션을 보면, **Then** 모든 오버레이가 동일한 페이드인/아웃 트랜지션을 사용한다

---

### Edge Cases

- 저사양 기기에서 우주 장식 요소가 과도하면 프레임 드롭이 발생할 수 있다 — 장식 요소 수를 디바이스 성능에 따라 조절해야 한다
- 오버레이가 열린 상태에서 뒤의 3D 씬 인터랙션(스크롤, 클릭)이 차단되어야 한다
- OAuth 콜백 처리 중 오버레이 상태가 유지되어야 한다 (페이지 리로드 시 오버레이 재표시)
- 극단적 화면 크기(4K 이상, 320px 미만)에서 오버레이와 장식 요소가 정상 표시되어야 한다
- 3D WebGL 미지원 브라우저에서 장식 요소가 그레이스풀하게 숨겨져야 한다

## Requirements _(mandatory)_

### Functional Requirements

**우주 장식 요소**

- **FR-001**: 시스템은 우주 뷰 배경에 별 파티클 필드(starfield)를 표시하여 깊이감을 제공해야 한다
- **FR-002**: 시스템은 유성(meteor) 오브젝트를 불규칙 간격으로 생성하고 발광 꼬리(trail)와 함께 이동시켜야 한다
- **FR-003**: 시스템은 다양한 크기의 운석(asteroid) 오브젝트를 3D 공간에 배치하고 느린 궤도 운동을 시켜야 한다
- **FR-004**: 시스템은 블랙홀 오브젝트를 주변 빛 왜곡(distortion) 시각 효과와 함께 렌더링해야 한다
- **FR-005**: 시스템은 태양(발광체) 오브젝트를 주변 조명 효과와 함께 배치해야 한다
- **FR-006**: 모든 장식 요소는 사용자 데이터 오브젝트(행성, 은하)와 겹치지 않는 위치에 배치되어야 한다

**UI 컴포넌트 전환**

- **FR-007**: 모든 버튼, 입력 필드, 텍스트 영역은 shadcn/ui 컴포넌트로 교체되어야 한다
- **FR-008**: 모든 카드, 다이얼로그, 선택(select) UI는 shadcn/ui 컴포넌트로 교체되어야 한다
- **FR-009**: shadcn/ui 테마는 우주 테마(어두운 배경, 연보라 악센트)에 맞게 커스터마이징되어야 한다

**스타일 마이그레이션**

- **FR-010**: 모든 UI 컴포넌트의 인라인 스타일(`style={{...}}`)은 Tailwind CSS 유틸리티 클래스로 전환되어야 한다
- **FR-011**: Three.js Canvas 자체 속성(width, height 등 WebGL 필수 설정)을 제외한 모든 스타일은 Tailwind로 관리되어야 한다

**오버레이 시스템**

- **FR-012**: 로그인/회원가입은 별도 페이지가 아닌 현재 화면 위의 오버레이 모달로 처리되어야 한다
- **FR-013**: 오버레이는 연보라 계열 반투명 배경과 글래스모피즘 효과를 가져야 한다
- **FR-014**: 모든 오버레이(로그인, 게시글 작성, 은하 생성)는 공통 오버레이 컴포넌트를 사용해야 한다
- **FR-015**: 오버레이는 배경 클릭 또는 ESC 키로 닫을 수 있어야 한다
- **FR-016**: 오버레이 열기/닫기에 부드러운 페이드 애니메이션이 적용되어야 한다

### Key Entities

- **장식 요소(Decoration)**: 우주 공간을 채우는 비인터랙티브 3D 오브젝트(유성, 운석, 블랙홀, 태양, 별 파티클). 유형, 위치, 크기, 움직임 패턴을 가짐
- **오버레이(Overlay)**: 현재 화면 위에 렌더링되는 반투명 모달 컨테이너. 배경 스타일, 콘텐츠 영역, 열기/닫기 상태를 가짐
- **테마(Theme)**: shadcn/ui 및 Tailwind CSS 변수로 정의되는 색상/간격/타이포그래피 체계. 우주 테마 특화 값을 가짐

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 사용자가 은하 보드 진입 시 최소 4종의 장식 요소(별 파티클, 유성, 운석, 블랙홀/태양)가 화면에 표시된다
- **SC-002**: 장식 요소가 추가된 상태에서도 일반 디바이스에서 초당 30프레임 이상의 렌더링 성능을 유지한다
- **SC-003**: 프로젝트 전체에서 UI 컴포넌트의 인라인 스타일(`style=`) 사용이 Three.js Canvas 설정을 제외하고 0건이다
- **SC-004**: 모든 폼/버튼/입력 컴포넌트가 shadcn/ui 기반으로 통일되어 시각적 일관성을 갖춘다
- **SC-005**: 로그인 플로우가 페이지 이동 없이 오버레이 내에서 완료되며, 로그인 완료까지 클릭 수가 기존 대비 동일하거나 감소한다
- **SC-006**: 모든 오버레이(로그인, 게시글 작성, 은하 생성)가 동일한 공통 오버레이 컴포넌트를 사용하여 디자인이 통일된다

## Assumptions

- 현재 Tailwind CSS v4 및 관련 설정(postcss, globals.css)이 이미 프로젝트에 구성되어 있다
- shadcn/ui 초기 설치 및 설정은 이 피처 범위에 포함된다 (현재 빈 디렉토리만 존재)
- Clerk 인증 시스템은 유지하되, 로그인 UI만 오버레이 방식으로 변경한다 (Clerk Custom Flow 사용)
- 장식 요소는 순수 시각적 요소로, 사용자 인터랙션(클릭 등)이나 백엔드 데이터와 연동되지 않는다
- 3D 장식 요소의 성능 최적화(LOD, instancing 등)는 이 피처 범위에 포함된다
- 모바일 반응형 대응은 오버레이에 한정하며, 3D 씬의 모바일 최적화는 별도 피처로 분리한다
- 기존 기능(게시글 작성, 댓글, 별 주기, 행성 커스터마이즈)의 동작은 유지하면서 UI만 교체한다
